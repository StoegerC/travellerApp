DATENHALTUNG
[ ] Selbst gehostetes Backend (Plan: ~/.claude/plans/lively-moseying-jellyfish.md) —
    Phase 1 läuft live: Server + SQLite + Endpunkt-Parität + Tailscale-Funnel-Tunnel +
    macOS-Autostart (launchd) stehen, echte Migration gegen den produktiven Worker
    gefahren, erstes Gerät umgestellt und Sync verifiziert. Noch offen: restliche Geräte
    umstellen, ein paar Tage beobachten, dann Cloudflare Worker/KV abschalten.
    Phase 2 (Datei-Uploads) und Phase 3 (Nutzerverwaltung/Login) sind umgesetzt, siehe
    CHANGELOG.md. Noch offen aus Phase 3: alle Geräte (iPhone, weitere Browser,
    GitHub-Pages-Zugriff) müssen sich einmalig neu mit E-Mail+Passwort anmelden, da der
    alte gemeinsame API-Key beim Cutover sofort ungültig wurde.
[ ] Charakter per QR-Code oder Link teilen (Base64-JSON)
[ ] Rollback (Admin-Seite, Snapshot wiederherstellen) sauberer gegen gleichzeitige
    Geräte-Pushes absichern — aktuell kann ein Gerät mit noch nicht synchronisierten
    lokalen Änderungen einen Rollback beim nächsten eigenen Speichern teilweise wieder
    rückgängig machen (Sync-Modell ist "lokal gewinnt bei Push-Konflikt"). Admin-UI weist
    nur mit einem Hinweistext darauf hin, löst es aber nicht strukturell.
